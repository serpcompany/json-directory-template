# Rebrand Checklist

<p className="mt-2 text-xl text-muted-foreground">
  Use this checklist when turning the starter into your own brand. It covers the
  checked-in config, public content, assets, links, and legal copy.
</p>

## What to replace first

- Site name, domain, and tagline
- Social links and repository references
- Submit destination and GitHub issue settings
- Favicon, logo, and Open Graph image
- About and legal content
- Any leftover starter wording in public pages

## Checked-in config checklist

- [ ] Set `site.*` in `sites/<id>/site-config.ts`
- [ ] Set `copy.*`, `routes.*`, and `features.*` in `sites/<id>/site-config.ts`
- [ ] Review `social.*` in `sites/<id>/site-config.ts`
- [ ] Review `build.*` and any deploy settings in `sites/<id>/site-config.ts`
- [ ] Review `sites/<id>/site-content.ts` for optional outbound resources

## Asset checklist

- [ ] Add or replace site assets under `sites/<id>/assets/**`
- [ ] Wire `branding.favicon`, `branding.logo`, and `branding.opengraphImage` in `sites/<id>/site-config.ts`
- [ ] Review screenshots or public images in `apps/web/public/**` if they still show starter branding

## Product copy checklist

- [ ] Review `packages/content/data/about/about.mdx`
- [ ] Review `packages/content/data/docs/**`
- [ ] Review `packages/content/data/guides/**`
- [ ] Review `packages/content/data/resources/**`
- [ ] Review homepage section copy such as hero, FAQ, and footer text

## External links checklist

- [ ] Update `social.githubUrl`, `social.githubRepoUrl`, `social.githubIssuesUrl`, and issue-template settings
- [ ] Review any links surfaced on `/projects`
- [ ] Review any outbound entries in `sites/<id>/site-content.ts`

## Legal and contact checklist

- [ ] Review `packages/content/data/legal/privacy.mdx`
- [ ] Review `packages/content/data/legal/terms.mdx`
- [ ] Review `packages/content/data/legal/cookies.mdx`
- [ ] Replace domains, emails, company/operator names, and contact details
- [ ] Update the “Last updated” date whenever legal copy changes

## Runtime details worth deciding

- [ ] Decide whether to rename the favorites storage key for your production brand
- [ ] Decide whether any optional routes should stay off by default for your site
- [ ] Decide whether internal package names need to change now or can stay internal until later

## Do not bulk-replace blindly

- `data/listings.json` contains third-party names that should stay intact
- Some screenshots may need to be re-recorded rather than string-replaced
- Example links in docs and guides may be intentional placeholders, so verify them before changing

## Suggested order

1. Replace site config values and shell links
2. Replace public content and legal copy
3. Replace visual assets
4. Run one final string sweep for starter residue
5. Build and manually verify the public pages

## Fast search commands

```bash
rg -n "Directory Starter|json-directory-template|serpcompany|example.com" apps/web packages/content sites
```

```bash
rg -n "privacy@|legal@|support@|contact@" packages/content/data/legal apps/web sites
```

## Verification checklist

- [ ] The homepage title and description use the new brand
- [ ] The header and footer show the new brand and external links
- [ ] The favicon and Open Graph image are replaced
- [ ] `/about`, `/legal/privacy`, `/legal/terms`, and `/legal/cookies` match the new brand
- [ ] `/submit` points to the correct repository and issue flow
- [ ] `/docs/getting-started` reflects your current site setup
- [ ] `pnpm build:site -- --site <id>` passes
