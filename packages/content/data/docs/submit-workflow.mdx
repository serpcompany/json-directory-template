# Submit Workflow

<p className="mt-2 text-xl text-muted-foreground">
  The active starter keeps submissions simple. The site gathers the basics, then
  sends the user to a prefilled GitHub issue for review.
</p>

## Why the starter uses GitHub issues

The current starter does not run a submissions backend. It stays static-friendly by using GitHub as the review queue.

That means the active app does not need:

- Site auth
- A database
- A server action or API route for submissions
- A moderation dashboard in the app itself

Instead, the repo becomes the source of truth and GitHub becomes the audit trail.

## What happens when someone submits

1. The user opens `/submit`.
2. The form collects the listing name, category, website URL, description, and any optional notes.
3. When the user clicks **Continue on GitHub**, the client builds a GitHub issue URL.
4. The browser navigates to the GitHub issue composer with a template, title, and prefilled body.
5. A maintainer reviews the issue and, if approved, updates repo data such as `data/listings.json`.
6. If that accepted change becomes a PR, CI validates `data/listings.json` before merge.

## Current category rule

The public form currently collects one category only.

Treat that value as the primary category for intake.
If the listing needs secondary categories, maintainers add them later during review in the checked-in source data.

## Example GitHub issue URL

```txt
https://github.com/OWNER/REPO/issues/new
  ?template=submit-website.yml
  &title=Submit listing: Example Project
  &body=...
```

## Example prefilled issue body

```md
## Listing details

Name: Example Project
Website: https://example.com
Category: developer-tools

## Description

A short summary of what the project does.

## Additional notes

Helpful reviewer context.
```

## Files that control the flow

- `apps/web/app/submit/page.tsx` renders the public submit page.
- `packages/web-core/src/forms/github-issue-submit-form.tsx` renders the form and handles the client-side redirect.
- `apps/web/lib/github-issue.ts` builds the final GitHub issue URL, template, title, and body.
- `sites/site-config.default.ts` plus `sites/<id>/site-config.ts` provide the checked-in owner, repo, issue URL, and template values.
- `.github/workflows/update-listings-json.yml` validates accepted listing changes when `data/listings.json` is edited in a PR or on `main`.
- `docs/knowledge/taxonomy-discovery-contract.md` records the current shared starter taxonomy and single-category intake rule.

## How accepted listing changes are gated

The current starter treats GitHub issues as intake and pull requests as the write path.

That means:

- the public site does not write listing JSON directly
- maintainers review submissions in GitHub first
- any accepted edit to `data/listings.json` must pass the dedicated listing-data validation workflow before merge

## What to change if you customize it

- Change the GitHub owner, repo, or template in the checked-in site config.
- Add or remove fields in `packages/web-core/src/forms/github-issue-submit-form.tsx`.
- Keep the GitHub issue handoff if you want to stay static-friendly with no runtime infrastructure.

If you also need the active data contract for accepted entries, see [JSON Shapes](/docs/json-shapes).
