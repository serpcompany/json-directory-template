# JSON Shapes

<p className="mt-2 text-xl text-muted-foreground">

## `data/listings.json`

This file is an array of listing entries.

```json
[
  {
    "slug": "example-project",
    "name": "Example Project",
    "website": "https://example.com",
    "description": "Short summary of the project.",
    "category": "developer-tools",
    "favicon": "https://www.google.com/s2/favicons?domain=example.com&sz=128",
    "featured": true,
    "resourceLinks": [
      {
        "label": "Help Center",
        "url": "https://example.com/help"
      }
    ],
    "publishedAt": "2026-03-22"
  }
]
```

### Website entry fields

| Field           | Required    | Notes                                                                                            |
| --------------- | ----------- | ------------------------------------------------------------------------------------------------ |
| `slug`          | Recommended | Stable URL segment for `/listing/[slug]`. The loader can derive it from `name` if it is missing. |
| `name`          | Yes         | Display name used in cards, search, and detail pages.                                            |
| `website`       | Yes         | Canonical site URL. The loader also accepts the legacy `domain` field.                           |
| `description`   | Yes         | Plain text summary. The loader strips HTML if older content included markup.                     |
| `category`      | Yes         | Category slug used by the starter UI.                                                            |
| `featured`      | No          | Marks the entry for featured lists.                                                              |
| `favicon`       | No          | Generated convenience field. The current UI does not require it to render.                       |
| `resourceLinks` | No          | Optional label/URL pairs rendered under the listing detail Resources section.                    |
| `content`       | No          | Optional body content rendered on the listing detail page.                                       |
| `priority`      | No          | Optional internal ordering hint used by some list views.                                         |
| `isUnofficial`  | No          | Marks community-maintained or non-official entries.                                              |
| `publishedAt`   | Yes         | Date string used for sorting and â€śrecently addedâ€ť sections.                                      |

## Loader compatibility rules

The website loader currently accepts a few migration-friendly variations:

- `website` or `domain`
- Missing `slug`, as long as `name` is present

The loader also:

- Sanitizes descriptions into plain text
- Sorts entries by `publishedAt`, newest first
- Maps the JSON entry into the active `WebsiteMetadata` shape used by the app

## Submit form payload shape

The `/submit` form does not write JSON directly. It builds a GitHub issue URL from this client-side shape:

```json
{
  "name": "Example Project",
  "website": "https://example.com",
  "category": "developer-tools",
  "description": "Short summary of the project.",
  "logoUrl": "https://example.com/logo.png",
  "videoUrl": "https://www.youtube.com/watch?v=abc123",
  "faqs": [
    {
      "question": "Does it work in Chrome?",
      "answer": "Yes."
    }
  ],
  "resourceLinks": [
    {
      "label": "Docs",
      "url": "https://example.com/docs"
    }
  ],
  "notes": "Helpful reviewer context."
}
```

That payload is converted into a prefilled GitHub issue. See [Submit Workflow](/docs/submit-workflow) for the full handoff.

The current public intake rule stays single-category.
If a listing needs secondary categories, maintainers add them later during review in the checked-in source of truth.

## Operator note

- `data/listings.json` is only for the main listing collection.
- Docs, posts, legal pages, and future first-party `/tools` content live elsewhere.
- If your site uses a source adapter like `sites/<id>/products.json`, build that source into `data/listings.json` before export.

## Other content shapes

The starter is JSON-first for the directory, but docs and guides still use MDX frontmatter.

### Docs frontmatter

```md
---
title: 'Page title'
description: 'Short summary'
order: 1
published: true
---
```

### Guides frontmatter

```md
---
title: 'Guide title'
description: 'Short summary'
date: '2026-03-22'
image: '/guides/example.webp'
authors:
  - name: 'Your Name'
difficulty: 'beginner'
category: 'getting-started'
published: true
---
```
