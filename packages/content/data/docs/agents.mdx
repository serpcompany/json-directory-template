# AI-Assisted Workflow

<p className="mt-2 text-xl text-muted-foreground">
  The starter works well with coding agents, but the source of truth still needs
  to stay in checked-in config, content, and data files.
</p>

## Recommended boundaries

Use these files for real product changes:

- `sites/<id>/site-config.ts` for site identity, routes, features, and GitHub handoff settings
- `sites/<id>/site-content.ts` for optional outbound resources or site-owned install mappings
- `data/listings.json` for listing entries
- `packages/content/data/**` for public docs, guides, about copy, and legal copy
- `docs/knowledge/**` for internal maintainer notes rather than public site content

## Suggested change loop

1. Inspect the current site config and data source before editing.
2. Make the smallest checked-in change that matches the product decision.
3. Run focused tests first, then a site build if the change affects output.
4. Verify the actual page, route, or artifact the user will see.
5. Record any new starter-specific rules in `docs/knowledge/**`.

## Browser verification

For UI work, run:

```bash
pnpm dev
```

Then check the page directly in a browser or with `curl` before calling the change done.

For static output, build one site and inspect the generated artifact:

```bash
pnpm build:site -- --site your-site-id
```

## Good agent tasks in this starter

- Cleaning up starter wording or metadata
- Updating site config fields and route labels
- Refactoring generic components without changing behavior
- Rewriting public docs to match the current product decisions
- Running focused tests and artifact checks after a change

## Risky tasks that need extra care

- Renaming compatibility fields without a migration path
- Changing listing schema without updating generators and tests
- Editing checked-in site config and public content in different directions
- Assuming optional routes are live without verifying the current feature flags

## Keep public and internal docs separate

- Public `/docs` should explain how to use or customize the starter.
- Internal operator notes belong in `docs/knowledge/**`.
- If a note is only for maintainers, do not leave it as public `/docs` content.
